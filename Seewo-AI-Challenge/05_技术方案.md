> **设计理念**：传统"管道式"架构（OCR→判定→统计）的缺陷在于每个环节只能看到上游的输出，无法灵活调度。**Agent 架构**的核心差异在于——每个 Agent 是一个有自主决策能力的单元，Orchestrator 可以根据输入特征动态组合 Agent 的调用路径：一道简单的选择题可能只需 OCR→Math Reasoning 两步；一道复杂的含参讨论题可能需要 OCR→Math Reasoning→Knowledge Graph→Diagnosis→Teaching Strategy 五步协作。

### 2.2 各 Agent 职责边界

| Agent | 一句话职责 | 触发条件 | 依赖 |
|-------|-----------|---------|------|
| **OCR Agent** | 手写体→结构化文本+公式→LaTeX | 任何含图片的提交 | Vision LLM |
| **Math Reasoning Agent** | 解答题步骤级验证（对/错/部分对+理由） | 检测到解答题 | OCR Agent 输出 |
| **Knowledge Graph Agent** | 题目→知识点标签；错题→知识薄弱点关联 | 批改完成后 | 题库+知识图谱 |
| **Diagnosis Agent** | 聚合多维度学情数据→归因分析→学生画像 | 单次作业批改完成后 | KG Agent + Math Agent |
| **Teaching Strategy Agent** | 基于诊断结果→推荐教学重点与分层策略 | 教师查看学情时 | Diagnosis Agent |
| **Lesson Planning Agent** | 生成课堂讲评PPT+讲评重点排序 | 教师请求讲评材料时 | Diagnosis + Strategy |
| **Exercise Generation Agent** | 匹配变式题+按学生水平分层推送 | 订正完成后 / 教师手动 | KG Agent + 题库 |
| **Classroom Assistant Agent** | 将讲评材料推送到希沃白板、实时互动 | 教师进入课堂模式 | Lesson Agent + 希沃白板API |

## 3. Agent 规格定义

> 采用Agent Card标准模板，每个智能体明确定义：角色、目标、输入/输出、工具、知识库、记忆、决策逻辑、协作关系、失败处理、评价指标。

### 3.1 OCR Agent（识别智能体）

| 维度 | 定义 |
|------|------|
| **Role** | 将学生手写作业照片转化为结构化文本和数学表达式 |
| **Goal** | 识别准确率：手写中文≥95%、数学公式≥92%（常见）≥85%（复杂） |
| **Workflow** | 图像预处理 → 版面分析 → 文本行检测 → 手写文字识别 → 公式区域定位 → LaTeX转换 → 结构化输出 |
| **Input** | 学生作业照片（JPEG/PNG，≥200dpi）+ 预期题目结构 |
| **Output** | `{text_blocks[], formula_blocks[{latex, confidence, bbox}], layout_tree}` |
| **Tools** | 图像预处理工具（OpenCV）、PaddleOCR、通义千问VL（公式识别fallback） |
| **Knowledge** | 高中数学符号集、常见手写变体字典、学科版面模板（横排/竖排/混排） |
| **Memory** | 同一学生的笔迹特征（持续学习，提高后续识别率） |
| **Decision** | 置信度>0.95：直接采纳；0.7-0.95：标记低置信度，传递给下游Agent附带置信度标注；<0.7：拒绝识别，提示"请重新拍摄清晰照片" |
| **Collaboration** | 向 Math Reasoning Agent 输出结构化解答文本；接收 Teacher 的修正反馈以优化模型 |
| **Fallback** | 对低质量图片，先尝试图像增强（去阴影/锐化），仍失败则引导用户重拍 |
| **Metrics** | 字符识别准确率、公式LaTeX转换正确率、端到端一次识别成功率 |

### 3.2 Math Reasoning Agent（数学推理智能体）

| 维度 | 定义 |
|------|------|
| **Role** | 对数学解答题进行步骤级推理验证，判定每一步的正确性 |
| **Goal** | 步骤正误判定准确率≥85%，错因分类准确率≥80% |
| **Workflow** | 接收OCR输出 → 题目语义理解 → 标准答案步骤模型构建 → 学生步骤切分对齐 → 逐步三维验证（逻辑×计算×规范）→ 错因归类 → 置信度评分 |
| **Input** | OCR输出的结构化解答 + 题目标准答案 + 评分标准 |
| **Output** | `{step_results[{step_num, correct, error_type, comment, suggested_fix}], overall_score, confidence}` |
| **Tools** | SymPy（符号计算验证）、WolframAlpha API（复杂计算核对）、LaTeX解析器、题库相似题检索 |
| **Knowledge** | 高中数学定理库、常见错误模式库（500+典型错误）、解题策略模板 |
| **Memory** | 该学生历史错误类型分布，用于个性化错因预判 |
| **Decision** | 规则引擎处理可穷举判定（如求导公式正误、二次方程求解）；LLM处理开放式推理（如证明逻辑是否严密、分类讨论是否完整）；置信度<0.7标记为"需教师复核" |
| **Collaboration** | 接收 OCR Agent 的文本；向 Diagnosis Agent 传递错因数据；向 Knowledge Graph Agent 查询知识点关联 |
| **Fallback** | 步骤切分失败→整段送LLM做整体判断；无法确定错因→归类为"待教师判断" |
| **Metrics** | 步骤判定准确率、错因分类准确率、教师修正率（目标≤10%） |

### 3.3 Knowledge Graph Agent（知识图谱智能体）

| 维度 | 定义 |
|------|------|
| **Role** | 维护学科知识图谱，将题目和错题映射到知识点体系 |
| **Goal** | 知识点自动标注准确率≥90%，知识薄弱点关联≥85% |
| **Workflow** | 题目文本→知识点向量检索→图谱匹配→多级标签标注→错题→薄弱点路径溯源→前置依赖知识点分析 |
| **Input** | 题目文本/LaTeX + 批改结果（正误+错因） |
| **Output** | `{knowledge_tags[], prerequisite_chain[], weakness_analysis{knowledge_point, mastery_estimate, evidence}}` |
| **Tools** | Milvus向量检索、Neo4j知识图谱查询、Embedding Model |
| **Knowledge** | 高中数学课标知识图谱（6大模块/32个单元/150+知识点，含前置依赖关系） |
| **Memory** | 全年级学生的知识点掌握度矩阵（实时更新） |
| **Decision** | 知识点标注：向量相似度+图谱规则双校验；薄弱分析：连续3次同类错误→标记为"待强化"；单一错误→标记为"需关注" |
| **Collaboration** | 向 Math Agent 提供题目的知识标签；向 Diagnosis Agent 提供薄弱点关联；向 Exercise Agent 提供知识点→题库映射 |
| **Fallback** | 无法匹配到精确知识点→上溯到父级知识点；新题型→人工标注+系统学习 |
| **Metrics** | 知识点标注准确率、薄弱点预测与教师判断的一致性 |

### 3.4 Diagnosis Agent（学情诊断智能体）

| 维度 | 定义 |
|------|------|
| **Role** | 聚合多源学情数据，生成学生/班级/年级的多维度诊断报告 |
| **Goal** | 取代教研组长手工Excel统计，将数据分析时间从8-12小时/月降至分钟级 |
| **Workflow** | 收集本次批改结果 → 合并历史学情数据 → 多维度聚合（学生/知识点/班级/年级）→ 归因分析 → 异常检测 → 生成可视化报告 |
| **Input** | Math Agent的批改结果、KG Agent的知识点标注、历史学情数据库 |
| **Output** | 知识热力图、学生雷达图、班级对比矩阵、异常预警、归因叙事 |
| **Tools** | SQL聚合查询引擎、ECharts可视化库、统计异常检测算法 |
| **Knowledge** | 教育测量学指标（难度系数、区分度、信度）、学情分析框架 |
| **Memory** | 全维度学情快照（周期T+1更新），支持时间序列趋势分析 |
| **Decision** | 某知识点正确率<60%→自动预警并上溯；某班某知识点突降>20%→标记异常；学生连续下降→生成关注提醒 |
| **Collaboration** | 向 Teaching Strategy Agent 传递教学建议输入；向 Lesson Agent 传递讲评重点；向 Exercise Agent 传递需巩固的知识点 |
| **Fallback** | 数据不足（新学校/新学期）→降级为基础统计，标注"数据积累中" |
| **Metrics** | 分析结论与教师主观判断的一致性、预警的准确率和召回率 |

### 3.5 Teaching Strategy Agent（教学策略智能体）

| 维度 | 定义 |
|------|------|
| **Role** | 基于诊断结果，为教师推荐教学策略和分层方案 |
| **Goal** | 将"多练""抓基础"等模糊建议转化为可操作的具体教学动作 |
| **Input** | Diagnosis Agent的归因报告 + KG Agent的知识点依赖链 |
| **Output** | `{priority_topics[], suggested_approach[], grouping_suggestion{}, time_allocation{}}` |
| **Tools** | LLM推理、教学策略模板库 |
| **Knowledge** | 教育学策略库（支架式教学/变式训练/概念转变等）、分层教学模型 |
| **Decision** | 班级整体薄弱→建议集中讲评+变式巩固；少数学生薄弱→建议课后分组辅导；前置知识缺失→建议回溯讲解 |
| **Collaboration** | 向 Lesson Agent 和 Exercise Agent 传递策略参数 |
| **Metrics** | 策略建议被教师采纳率 |

## 5. 核心技术选型

| 层次 | 技术 | 选型理由 |
|------|------|---------|
| 前端·教师端 | React + TypeScript | 希沃内部技术栈一致，组件复用 |
| 前端·学生端 | 微信小程序原生 | 学生覆盖率最高，无需安装APP |
| 前端·教研端 | React + ECharts | 复杂图表交互需求 |
| 后端·API | Python FastAPI | AI生态最完善（LLM/ML SDK），异步性能好 |
| **Agent框架** | **LangGraph + 自研 Orchestrator** | LangGraph 提供状态图编排；自研层处理教学场景特化逻辑（如置信度路由、教师反馈回路） |
| 数据库 | PostgreSQL + Milvus | 业务数据（PG JSONB）+ 知识点向量检索（Milvus） |
| 缓存/队列 | Redis + Celery | 批改任务异步队列 + 热点缓存 |
| OCR | PaddleOCR + 通义千问VL | 通用OCR（本地）+ 公式/图表（视觉大模型 fallback） |
| 推理LLM | DeepSeek-Math + 通义千问 | 数学推理优先用DeepSeek-Math，通用评语/分析用千问 |
| 知识图谱 | Neo4j | 知识点之间的前置依赖→后继延伸的图查询天然高效 |
| 部署 | Docker + K8s | 公有云SaaS / 混合云 / 私有化三种模式 |

## 9. 安全与合规

- **等保三级**：网络安全、主机安全、应用安全、数据安全全覆盖
- **数据加密**：TLS 1.3（传输）+ AES-256（存储）
- **权限模型**：RBAC 五级（超管→校管→教研组长→教师→学生）
- **Agent 权限隔离**：每个 Agent 仅有完成其任务所需的最小数据访问权限
- **审计日志**：全量数据访问 + Agent决策轨迹记录，保存≥180天，不可删除
- **个人信息保护**：学生姓名默认脱敏，支持数据导出/删除/撤回同意
- **部署灵活性**：公有云SaaS（默认）/ 混合云 / 本地化私有部署
